# peaks-loop slice 审阅 + QA 视角

> 适用: 当 LLM 解释 Code gate / slice review / QA 验收时,避免让 user 审技术。

## 1. 核心原则

**user 在循环里 = 业务/产品审阅者,不是技术验收者。**

- ✅ 该看:业务流程、需求覆盖、边界 case、UI 装配、异常态语调、能上线吗
- ❌ 不看:覆盖率、性能、安全、单元测试、E2E 报告、库选、设计质量

## 2. Slice Review 模板(每个 slice 完成,user 必审 4-5 项)

```
Slice 业务验收清单:

  1. 这个 slice 做完,业务流程对吗?
     - 跟产品最初给的需求匹配吗?
     - 跟现有系统交互有没有断层?

  2. 老板提的需求都覆盖了吗?
     - 大需求点 vs 小需求点
     - 显式需求 vs 隐式需求

  3. 边界 case 跟业务预期一致吗?
     - 异常输入下的引导
     - 错误提示的用户语言
     - 空状态 / 加载状态 / 失败状态的产品语调

  4. UI 装配跟产品预期一致吗?
     - 页面模式 / 关键交互 / 信息密度 是否符合产品预期
     - 异常态 / 边界态视觉跟产品语调一致吗

  5. 能合入下一个小版本吗?
     - 业务风险
     - 用户体验风险
     - 跟其他 slice 集成有断点吗
```

## 3. QA 业务视角验收(6 项,全产品使用视角)

```
QA 业务验收清单:

  1. 这个功能"用起来"对吗?
     - 业务流程顺不顺?
     - 操作路径是否反人类?
     - 跟现有系统交互有没有断层?

  2. 老板提的需求都覆盖了吗?
     - 大需求点 vs 小需求点
     - 显式需求 vs 隐式需求

  3. 边界 case 跟业务预期一致吗?
     - 异常输入下的引导
     - 错误提示的用户语言
     - 空状态 / 加载状态 / 失败状态的产品语调

  4. 页面模式 / 关键交互 / 信息密度 跟产品预期一致吗
     ← "装配验收",user 真正擅长的部分

  5. 异常态 / 边界态视觉跟产品语调一致吗
     - 空状态:有友好引导
     - 加载态:不闪烁
     - 失败态:有重试入口
     ← "用户体验语调",不是"设计质量"

  6. 能合入下个小版本吗?
     - 业务风险 / 用户体验风险 / 集成断点
```

## 4. QA 双层验证(peaks-qa 内部)

| 层 | 谁负责 | 视角 | user 介入? |
|---|---|---|---|
| **技术验证层** | peaks-qa 子代理 | 覆盖率、性能、安全、自动化 | ❌ AI 自决 |
| **业务验证层** | user | 业务、产品、边界 case、UI 装配 | ✅ user 必审 |

→ **关键技术点:**
- 技术验证层跑通 = slice 通过的硬条件(没跑通 = sub-agent 失败 = 退化串行,user 不参与决策)
- 业务验证层 = slice 完成的 user 介入点(user 必审,审过才进 final)
- 两者解耦:**技术不过 → AI 内部重跑;业务不过 → user 反馈修**

## 5. user 不看的内容(技术指标由 AI 内部消化)

- × RBAC 单元测试覆盖率
- × PG 迁移脚本幂等性测试
- × Skill 沙箱安全性测试
- × JWT 签名验证测试
- × 性能 P99 / 性能曲线
- × 安全扫描结果
- × 代码 diff
- × 单元测试报告
- × E2E 报告

## 6. 上游同步 slice 的 QA 业务验收(5 项,扩展)

```
1. 老 TOC 用户能正常用吗?(兼容性)
2. 我的定制功能(Skill 权限 / SkillHub / 鉴权)正常吗?
3. 上游新功能是否按预期工作?
4. 性能是否退化?(上游改动可能引入)
5. 同步后能合入下个业务版本吗?
```
