## D-013 fix details

**Bug:** `peaks <unknown>` and `peaks <unknown> --help` returned exit 0 with the help banner. Root cause: Commander's `.exitOverride()` was set but the catch in `src/cli/index.ts:12-15` swallowed `commander.helpDisplayed` exit codes, AND the root `.action()` in `src/cli/program.ts:211` unconditionally printed the banner with no exit code adjustment.

**Fix:** (commit `8145f01`)
- `src/cli/program.ts` root `.action()` now inspects `program.args` for a non-option positional token; if present, emits `COMMAND_NOT_FOUND` JSON envelope + `process.exitCode = 1`.
- `src/cli/index.ts` pre-check uses `setImmediate` to handle the `peaks <unknown> --help` case where Commander's help short-circuit fires BEFORE the unknownCommand throw.
- `src/cli/index.ts` catch block emits a structured `COMMAND_NOT_FOUND` envelope for `commander.unknownCommand`/`unknownOption` (LLM-side consumers).
- `tests/unit/cli/d-013-wrapper-exit-code.test.ts` (NEW): 5 regression cases.

**Critical detail:** use `program.args`, NOT `process.argv`. In vitest in-process integration tests, `process.argv` is the test runner's argv (empty), not the peaks CLI argv. `program.args` is consistent across both real-CLI and in-process paths.
