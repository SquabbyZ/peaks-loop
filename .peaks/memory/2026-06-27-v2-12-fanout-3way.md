# v2.12.0 RD Fan-out 3-way Shape (slice 2026-06-27)

## Decision

`src/services/rd/reviewer-dispatch-policy.ts` `RD_FANOUT_REVIEWERS` is
exactly the **3-element tuple** (in order):

```ts
export const RD_FANOUT_REVIEWERS = [
  'code-reviewer',
  'qa-test-cases-writer',
  'karpathy-reviewer',
] as const;
```

`security-reviewer` + `perf-baseline-reviewer` are NOT in the fan-out.
They are exposed as `RD_DEPRECATED_REVIEWERS` for the 1-minor-release
back-compat window:

```ts
export const RD_DEPRECATED_REVIEWERS = [
  'security-reviewer',
  'perf-baseline-reviewer',
] as const;
```

The `isDeprecatedReviewer(name)` predicate routes any legacy dispatch
record to the new audit skill instead of failing the gate.

## Why (the rationale)

Prior 5-way fan-out (slice 002 3-way + slice 004 qa-test-cases +
slice 5/6 karpathy-reviewer) conflated two concerns:

- **RD-internal reviews** (code, qa-test-cases, karpathy) — these are
  RD's own review surface; they belong in the RD fan-out.
- **Project-level audits** (security threat model, perf baseline) —
  these are stable across slices; they belong as standalone audit
  skills that consume the immutable handoff.

The v2.12.0 collapse separates these concerns. The RD fan-out now
contains exactly the RD-internal reviews. The project-level audits
become independent skills (`peaks-security-audit` / `peaks-perf-audit`)
whose outputs are mechanically enforced by prereqs.

## What stayed the same (still in the fan-out)

| Sub-agent | Output path | Why |
|---|---|---|
| `code-reviewer` | `rd/code-review.md` | RD's view of code quality |
| `qa-test-cases-writer` | `qa/test-cases/<rid>.md` | Pre-drafts QA test plan so QA main loop can skip "draft from scratch" |
| `karpathy-reviewer` (hard gate) | `rd/karpathy-review.md` | Enforces the 4 Karpathy-guidelines (Think Before Coding / Simplicity First / Surgical Changes / Goal-Driven Execution) |

## What moved out

| Slot | New home | Why |
|---|---|---|
| `security-reviewer` | `peaks-security-audit` (skill id) → `audit/security.md` | Standalone skill consumes handoff + audit template; project-level threat model is stable across slices |
| `perf-baseline-reviewer` | `peaks-perf-audit` (skill id) → `audit/perf.md` | Standalone skill consumes handoff + audit template; perf baseline is stable across slices |

## Anti-patterns (do NOT do)

- Do NOT add a 4th sub-agent to `RD_FANOUT_REVIEWERS` without also
  removing something. The 3-element shape is part of the public
  contract; adding to it breaks the fan-out dispatcher.
- Do NOT re-add `security-reviewer` or `perf-baseline-reviewer` to the
  fan-out. They live in `RD_DEPRECATED_REVIEWERS` for the
  1-minor-release window; the new audit skills replace them.
- Do NOT change the `RD_FANOUT_REVIEWERS` order. The order is the
  dispatcher order; tests + dispatcher consumers depend on it.
- Do NOT change the sub-agent slot file paths. Gate C prereqs + the
  `KARPATHY_REVIEW` prereq read the canonical paths.
