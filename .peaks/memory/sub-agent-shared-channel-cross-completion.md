User hard rule (2026-06-07 1:46 GMT+8): "**而且只要有子 agent 完成就更新 dispatcher 的共享上下文,这样其他未完成的子 agent 也能感知**". This is the **cross sub-agent shared channel** red line. It is orthogonal to G1..G7; registered as G8 in slice #009 PRD.

## Why

The pseudo-swarm model (G1 / pseudo-swarm declaration) declares sub-agents collaborate via **shared artifact state, no peer-to-peer message**. This is correct as the default. But it leaves one gap: when sub-agent A finishes partway through, sub-agent B (still in flight) doesn't know — it must wait for the dispatcher to reduce the whole batch. B might re-do work A already did, or miss a blocker A found.

The user's fix: sub-agent A's completion **immediately writes a shared entry**; sub-agent B (still in flight) can **read shared entries** from sibling sub-agents and adapt. This is **not** peer-to-peer messaging (sub-agent A doesn't directly send to B); it's **indirect signal via dispatcher-mediated shared channel** (A writes; dispatcher stores; B reads; A and B never directly talk).

## The rule (RL-23..RL-26 + G8 protocol)

**Path convention (G8.3)** — `.peaks/_sub_agents/<sid>/shared/<batchId>.json` (per-batch single file; R-2 guard consistent with dispatch record):
- `batchId` auto-generated by `peaks sub-agent dispatch` CLI; same batchId shared by all sub-agents in one batch
- File is atomic-write (tmp + rename) like dispatch record
- R-2 path guard: must be in `.peaks/_sub_agents/<sid>/shared/`
- GC at batch close (reducer complete); 30-day TTL on orphan channels

**SharedChannelEntry schema (G8.3)**:
```ts
interface SharedChannelEntry {
  readonly at: string;                    // ISO8601
  readonly from: string;                  // sub-agent role string (e.g. "rd", "qa-business")
  readonly key: string;                   // '<role>.<event>' convention (e.g. "rd.completed", "qa-business.found-blocker")
  readonly value: Readonly<Record<string, unknown>>;  // ≤ 1KB (soft warning); ≥ 64KB rejected
  readonly valueSize: number;             // bytes
}

interface SharedChannel {
  readonly batchId: string;
  readonly createdAt: string;
  readonly updatedAt: string;
  readonly entries: Readonly<Record<string, SharedChannelEntry>>;  // key → entry (last-write-wins)
}
```

**Two new CLI atoms (G8.4)** — added on existing `peaks sub-agent` namespace, no new top-level CLI:

```
peaks sub-agent share --batch <batchId> --key <key> --value <json> [--from <role>] --json
  --batch required, batchId string
  --key required, non-empty
  --value required, JSON object (CLI validates)
  --from optional, defaults to dispatch record's role
  --json required
  Returns: {ok, batchId, entryKey, writtenAt, channelSize}
  Errors: --key already exists (last-write-wins + warning), --value non-JSON, --value ≥ 64KB

peaks sub-agent shared-read --batch <batchId> [--since <iso>] [--key <pattern>] --json
  --batch required
  --since optional, ISO8601; only return entries written after this time
  --key optional, pattern (e.g. "rd.*" matches rd-prefixed keys)
  --json required
  Returns: {ok, batchId, entries: {[key]: SharedChannelEntry}, totalEntries, channelSize, updatedAt}
```

**RL-23 Completion-time mandatory write (strong constraint)**:
- When sub-agent calls `peaks sub-agent heartbeat --status done`, it MUST also call `peaks sub-agent share --key "<role>.completed" --value <artifact-meta>` (artifact-meta is the same ArtifactMeta from G7.4)
- If sub-agent omits the share, heartbeat still succeeds but emit warning `code: "COMPLETED_WITHOUT_SHARE"`
- Same level of constraint as G6 done heartbeat

**RL-24 Fire-and-forget (consistent with G6 RL-14)**:
- Share write is async, non-blocking
- Write failure → emit warning, do NOT retry, do NOT throw

**RL-25 Value size limit (anti-pattern prevention)**:
- value ≤ 1KB: soft warning
- value ≥ 64KB: rejected with `code: "VALUE_TOO_LARGE"`
- Prevents the anti-pattern "sub-agent pretends the whole artifact is a key"

**RL-26 batchId isolation**:
- Channel scope is `<sid> + <rid> + <batchId>` triple
- Cross-batch read rejected (same isolation as dispatch record)

**Sub-agent prompt template (G8.6 / G3 SKILL.md addition)**:
```
You are sub-agent role <role>, batch <batchId>.

PROTOCOL (mandatory):
1. On start: peek at shared channel: `peaks sub-agent shared-read --batch <batchId> --json`
   to see what other sub-agents in this batch have shared so far.
2. While running: if you find a blocker or partial work, write share entry
   `peaks sub-agent share --key "<role>.found-blocker" --value {"reason": "..."}`
   so other in-flight sub-agents can avoid duplicating effort.
3. On completion: write share entry
   `peaks sub-agent share --key "<role>.completed" --value <artifact-meta>`
   BEFORE the final `peaks sub-agent heartbeat --status done` heartbeat
   (or immediately after; either is fine).
4. The shared channel is your only visibility into sibling sub-agents.
   Do NOT attempt to read other sub-agents' dispatch records directly.
   (Shared channel is the contract; dispatch records are dispatcher-internal.)
```

## Pseudo-swarm model impact

G8 is a **partial upgrade** of pseudo-swarm property 3 (sub-agent collaboration):

| Pseudo-swarm property | Pre-G8 | Post-G8 |
|---|---|---|
| 3. Sub-agent collaboration | Indirect via shared artifact state, NO peer-to-peer | Indirect via shared artifact state + shared channel; STILL no peer-to-peer |

G8 adds **dispatcher-mediated shared channel** for sibling sub-agent status; it does NOT introduce peer messaging. The dispatcher remains the central reducer. Sub-agents still don't talk directly to each other.

## How to apply

For every sub-agent dispatch by peaks-solo / peaks-rd / peaks-qa:

1. Sub-agent prompt includes the G8.6 protocol block (mandatory)
2. SKILL.md fan-out sections reference `peaks sub-agent share` and `peaks sub-agent shared-read` CLIs
3. Dispatcher (peaks-solo main loop) creates the SharedChannel file at batch start
4. Dispatcher GCs SharedChannel at batch close (reducer complete)
5. Reducer emit view: "shared channel entries this batch" alongside the G7 metadata view

For dispatcher / Solo main loop:

1. On batch start: create `.peaks/_sub_agents/<sid>/shared/<batchId>.json` with empty entries
2. On batch close: emit SharedChannel summary alongside dispatch records; mark channel for GC
3. On session close / slice close: archive SharedChannels to `.peaks/_runtime/<sid>/_archive/_shared/<slice-id>/` (same as dispatch record archive)

## What does NOT satisfy the rule

- "Sub-agent A directly messages sub-agent B via some side channel" (violates pseudo-swarm property 3; not allowed)
- "Sub-agent A's `share` call goes to B's prompt directly" (same; the contract is "sub-agent writes; dispatcher stores; other sub-agent reads")
- "Sub-agent skips the share entry on done" (violates RL-23 — strong constraint, same level as G6 done heartbeat)
- "Sub-agent writes a 1MB blob as the value field" (violates RL-25 — value size limit prevents the anti-pattern)
- "Sub-agent reads other sub-agents' dispatch records directly" (violates G8.6 protocol — only the shared channel is the contract)

## Why this is additive, not a replacement

G5 (resource) / G6 (liveness) / G7 (content) / G8 (cross-completion) are four orthogonal axes:

| | G5 (resource) | G6 (liveness) | G7 (content) | G8 (cross-completion) |
|---|---|---|---|---|
| Concern | Create + dispose + reclaim | Run + heartbeat + stale | Artifact content occupation | Sibling sub-agent status |
| Question | Did we leak? | Is it alive? | Does main LLM explode? | Do in-flight sub-agents know what siblings finished? |
| Failure mode | Orphan records | User Ctrl-C | Main LLM silent-truncate | Sub-agent B redoes A's work |
| Mitigation | Record + reducer | Heartbeat + poller | Metadata-only + Read | Shared channel + share/shared-read |

All four must pass.
