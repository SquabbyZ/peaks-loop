## D-004 — `peaks sub-agent dispatch` 真实签名(SKILL.md 仍写错的)

**SKILL.md 4.0.0-beta.10 描述:** `--role / --task / --karpathy / --caller-id / --job-id / --slice-id`(待考)

**实测 (4.0.0-beta.10):**
- `<role>` **positional**(rd / qa / ui / txt / qa-business / qa-business-api)
- `--prompt <text>` (**不是** `--task`)
- `--request-id <rid>` (唯一 request 维度 flag, **不是** `--rid`)
- `--session-id <sid>` (**存在**,覆盖默认 session 解析)
- `--project <path>`
- `--batch-id <uuid>` (auto-generated by default)
- `--write-artifact <path>` (G7: 注册 artifact 文件)
- `--use-headroom` / `--headroom-mode <mode>` (G7.7)
- `--force` (G9 兜底,override 80% hard reject)
- `--from-dag <file>` (2.7.0 swarm,跟 `<role>` 互斥)
- `--json` (machine-readable)
- **没有**: `--task`, `--karpathy`, `--caller-id`, `--job-id`, `--slice-id`

**关键修正:** 4.0.0-beta.6 sediment 说"无 `--session-id`",**这条过时了** —— 4.0.0-beta.10 加回去了。

**另一条关键修正:** CLI 是 **dry-run** —— "LLM executes the returned toolCall in its own environment"。主 LLM 必须自己用 Agent 工具 dispatch,不能依赖 CLI 直接执行。

## D-005 — `peaks job checkpoint` 缺 `--evidence` flag

**RD tech-doc §Step 4 写的:**
```bash
peaks job checkpoint \
  --job-id 2026-07-16-cli-surface-cleanup-impl \
  --slice-id slice-1-del-minimax-worker \
  --state done \
  --commit-sha <sha> \
  --evidence .peaks/_runtime/2026-07-15-session-87a173/qa-evidence/slice-1.md
```

**实测 (4.0.0-beta.10) 真实签名:**
```
Options:
  --job-id <jid>
  --slice-id <rid>
  --state <done|failed|skipped>
  --commit-sha <sha>
  --reason <text>
  --project <repo>
  --json
```

**没有 `--evidence` flag。** QA evidence 文件必须 **手动写到磁盘**(本来就该如此),然后 peaks job checkpoint 只用 `--reason` 引用路径。

**Why this matters:** 如果照 SKILL.md 的命令跑,会被 `error: unknown option '--evidence'` 拒绝,然后 CLI 不会告诉主 LLM 该怎么修,只报 unknown option → 上下文里 sediment 没记录这条 → 下一个 session 又踩。

**How to apply:**
- 把 QA evidence 写到 `.peaks/_runtime/<sid>/qa-evidence/slice-<n>.md`(LLM 直接 Write)
- `peaks job checkpoint` 用 `--reason "qa-evidence: .peaks/_runtime/.../slice-<n>.md"` 引用
- 把 RD tech-doc §Step 4 的 `--evidence` 字段删除(下次 RD 写 tech-doc 时手动校对)

## D-006 — `peaks session title` 真实签名(sid positional 不是 flag)

**4.0.0-beta.6 sediment 说:** "sid 是 positional,不要 `--session-id` flag" — **部分过时**

**实测 (4.0.0-beta.10):**
```
Usage: peaks session title [options] <sessionId> <title>
Options:
  --json      print machine-readable JSON envelope
  -h, --help  display help for command
```

**正确:** `peaks session title <sessionId> "<title>" --json`(两个 positional: sid + title)
**错误:** `peaks session title --session-id <sid> "<title>" --project <path>` — `error: unknown option '--session-id'` + `error: unknown option '--project'`

4.0.0-beta.6 sediment 写对的是"positional",写错的是"`--session-id` flag 不存在" → 4.0.0-beta.10 还是这样。

**之前 4.0.0-beta.6 sediment 第二条:** "`--project` 不是合法 flag" → **仍正确**。

**Why this matters:** checkpoint 里我两次撞了这条 sediment 错误版本。**Sediment 不更新 = 反复撞**。

## D-004 + D-005 + D-006 共同教训

**SKILL.md + 4.0.0-beta.6 sediment 都过期了至少 1 个版本。** 长跑模式跑 rotate cycle 之前必须:

1. **新 session 第一个动作:** 跑 `peaks <cmd> --help` 把当前会话要用的所有 CLI 都打印一次,核对 SKILL.md + sediment 描述。
2. **CLI 改了 sediment 没改 → sediment 是陷阱。** 每次新 session 都要 `git grep -l 'peaks .* --<flag>' .peaks/memory/` 扫一遍可能的过期条目。
3. **CLI 真实签名变更必须立即更新到 sediment**,否则下游 session 反复撞同一条坑。

## 沉淀原则(从 4.0.0-beta.10 这次卡死得出)

1. **每次 CLI 跟 SKILL.md 不一致,不仅要 sediment 一条 drift,还要把"如果照错命令跑会得到什么 error"写进 sediment** —— 让下次 session 一看就知道怎么修。
2. **Drift ID 跟版本号挂钩**(D-001 起步于 4.0.0-beta.6,D-004 在 4.0.0-beta.10 仍有效 + 新增 D-005/D-006) —— 不要试图"统一编号",会让 sediment 失去版本对照。
3. **CLI 真实签名速查表是本 session 最有价值的产物**,比"D-XXX 单条规则"重要 10 倍。下次 session 第一件事就是读这个表。

Why: peaks-code 的 SKILL.md 和 sediment 跟真实 CLI 永远有滞后,长跑 rotating job 模式撞坑代价 = 主会话被卡死 + user 要手动 terminate。
How to apply: peaks-code 启动时 Step 0 之后立刻把"会话接下来要用的 CLI 速查表"打印出来贴到 context,避免撞 D-001/002/003/004/005/006 同款坑。
